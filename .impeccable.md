# Impeccable — Project Design Context

This file is the source of truth for design decisions on swpatterns.com. The `/impeccable` skills read it before doing any design work. Update it when the direction changes — don't let it drift.

---

## Design Context

### Users

The primary reader is a **working software engineer in lookup mode**. They arrived from a search engine with a specific pattern in mind (e.g. "saga orchestration", "circuit breaker"), they have a language they're working in, and they want two things in this order:

1. A short, authoritative definition that confirms they've found the right pattern.
2. A code sample in their language that they can read or copy.

A secondary audience exists — students browsing categories, practitioners cross-referencing related patterns — but every design decision is judged against the lookup case first. Anything that helps the lookup case (clearer hierarchy, faster language switching, scannable metadata) helps the others. Anything that gets in its way (illustrations, tutorial chrome, marketing voice) hurts everyone.

The site is content. It is not an app. Read at a desk, on a laptop, occasionally on a phone while pairing.

### Brand Personality

Three words: **archival, disciplined, technical-editorial.**

The site projects the voice of a printed reference manual — closer to an Olivetti technical catalogue or a Vignelli wayfinding system than a magazine cover. Authoritative without being academic. Dense without being cluttered. Italian without being kitschy (no flags, no tricolore stripes — Italian as in Olivetti, Vignelli, Lars Müller, Massimo Bellini; not as in pizza emojis).

Emotional goal: when a reader lands on a pattern page, they should feel they've arrived at something **considered**, not generated. The footer already says "Made with ❤️ in Bologna" — the design should earn that line.

### Aesthetic Direction

**Reference**: Italian editorial archival, technical-manual lineage. Specifically the visual language of **Olivetti's product documentation, Vignelli's NYC subway system, Lars Müller monographs, and the Bauhaus diagram tradition**. Page-as-artifact discipline. Pattern entries are typeset like the entries in a printed engineering reference: numbered, classified, deliberately set, in a workhorse type that holds up at 14pt.

This is a deliberate pivot from an earlier draft that leaned on **Bodoni-school editorial display** (Domus/Casabella magazine-cover energy). That direction proved too thin: hairline-stroke type doesn't render well at reading sizes on screen, and a saturated chromatic accent felt too vibrant for what should be a quiet reference. The new direction trades magazine-cover for technical-manual.

**Anti-references** (treat as hard exclusions):

- **No Refactoring.guru aesthetic**: no cartoon mascots, no color-coded category pills, no illustrated explainer tone.
- **No generic Hugo / dev-blog template**: no centered hero with a stock sans, no identical card grids, no drop-shadow rounded rectangles.
- **No AI-startup landing page**: no dark + cyan/purple gradients, no glassmorphism, no glowing borders, no gradient text.
- **No corporate enterprise docs**: no IBM/SAP navy-and-gray chrome, no dense nav rails that feel legal.
- **No magazine-cover energy**: no high-contrast Didone display type set huge in a saturated accent — that's the Domus/Vogue language and it conflicts with the technical-manual direction we're now committed to.

**Theme**: follow system preference (`prefers-color-scheme`) with a manual override toggle preserved. Light is **warm off-white** (book paper, not stock white). Dark is **deep ink** (not pure black; slight cool tint). Both must be designed to the same standard — no "we built it light and inverted it for dark."

**Color**: the palette is **near-monochrome ink on paper, no chromatic accent**. The Italian editorial vermillion the previous draft used as a single accent is gone — even one saturated highlight read as too vibrant for the reference brief. Emphasis comes from **weight, size, and rule lines**, never from hue.

Concrete starting palette in OKLCH (warm-tinted neutrals; everything tints toward ~75° at very low chroma):

- `--paper`:       `oklch(97.5% 0.005 75)` — page surface on light
- `--paper-dim`:   `oklch(94.5% 0.007 75)` — secondary surface on light
- `--paper-deep`:  `oklch(91% 0.008 75)` — code/aside surface on light
- `--ink`:         `oklch(20% 0.012 280)` — body text on light
- `--ink-soft`:    `oklch(40% 0.014 280)` — secondary text
- `--rule`:        `oklch(20% 0.012 280 / 0.18)` — hairline rules
- `--rule-strong`: `oklch(20% 0.012 280 / 0.36)` — heavier dividers
- (dark mode swaps surfaces and text via `light-dark()`; see `themes/archivio/assets/sass/_tokens.scss`)

The **only** place a chromatic note may appear is on a focused interactive element (focus ring) or on a clear "live indicator" — and even there, prefer a quiet ink-blue annotation tone (e.g. `oklch(40% 0.06 240)`) over a saturated red. **Catalog numerals, classification chips, language tabs, and link underlines are all monochrome — they get presence from weight or rule-line, not from color.**

If a future state genuinely needs a chromatic differentiator (e.g. an inline error ribbon), pick the most muted version of the Olivetti-pen-blue family rather than reaching for a red. Do not reintroduce the vermillion.

**Typography**: substantial, technical, not thin. Every font in the reflex-reject list is still banned. The new stack:

- **Display / pattern titles / catalog numerals** → **Archivo** (Omnibus-Type, OFL, Google Fonts). Variable, geometric grotesque named — by happy coincidence — for our theme. Holds at heavy weights (700–900) without feeling shouty; lining figures land cleanly in catalog numbers like `Nº 042`. Use weights 700–800 for `h1`, 600 for `h2`, never below 500 for headings.
- **Body / running text / `h2`–`h4`** → **Hanken Grotesk** (Hanken Co, OFL, Google Fonts). Workhorse neutral grotesque with substantial strokes — designed for reading at small sizes. Use weight 400 for body, 500 for emphasis, 600 for `h3`/`h4`. **No serif body face** — that was the previous draft's mistake.
- **Code** → **Anonymous Pro** (Mark Simonson, OFL, Google Fonts). Proper developer mono with slashed zero and distinct `I/l/1`. Stays from the previous iteration.
- **Labels / small-caps section marks** → **Hanken Grotesk in uppercase with positive tracking** (~0.10em). Drop the typewriter-mono label face from the previous draft — typewriter character conflicts with the technical-manual direction. Same family as body, distinguished only by case + tracking.

If any of these proves impractical, the substitution rule is: pick something that holds the same role and **is not** in the reflex-reject list. **Never** fall back to Inter, Roboto, Plex, Space Grotesk, Outfit, DM Sans, etc.

**Layout**: strong asymmetric grid. Left-aligned body, generous outer margins, distinct columns for content vs. metadata sidebar. Pattern pages read as numbered catalog entries — entry number top-left in **Archivo lining figures**, taxonomy classification (`Creational · GoF`) in Hanken small-caps with tracking, title in Archivo heavy, body in Hanken at 17px. Per-language code samples below, with a language switcher that is the **most prominent interactive element on the page** — that's why most readers came.

No card grids on the homepage. The pattern index is a typographic list — like a printed contents page — not a sea of identical tiles.

### Design Principles

1. **Catalog, not blog.** Every page is a numbered entry in an archive. Metadata (`pattern_types`, `language`, taxonomy) is composed into the design, not bolted on as chrome. Pattern types behave like a classification system, not tags.
2. **Lookup over discovery.** Assume the reader knows what they came for. Title, classification, and language switcher must all be visible without scrolling on a 13" laptop. Related-pattern navigation lives below the fold.
3. **Restraint, not ornament.** The design earns its character through typography weight, rule lines, and grid discipline — never through color, illustration, or decorative shapes. If something looks decorative, it is wrong.
4. **Code samples are the product.** Per-language code is what brought most readers here. Syntax highlighting must hit WCAG AA on both themes (verify, don't assume). Copy buttons must be keyboard-reachable. Code blocks must be screen-reader friendly (proper `<pre><code>` semantics, language declared). Highlighting may use one or two muted hues (deep blue, deep umber) but **not** the brand-removed vermillion.
5. **Dual-mode parity.** Light and dark are co-equal. Type weight, syntax-highlight contrast, and rule-line visibility are tuned independently for each. Test both before shipping any layout work.

### Constraints

- **Stack**: Hugo + LibSass (no Dart Sass on this CI; SCSS uses `@import` not `@use`, and any `clamp()` arithmetic outside CSS-vars must be wrapped in `calc()`). No JS frameworks. Style work lands in `themes/archivio/assets/sass/` or in `layouts/` overrides.
- **A11y**: WCAG **AA** baseline, plus extra-care clauses for code samples (AA syntax contrast, keyboard-reachable copy, screen-reader-friendly code blocks). Respect `prefers-reduced-motion`.
- **Performance**: no JS frameworks, no large icon fonts. Web fonts loaded with `font-display: swap`. Variable fonts preferred (Archivo is variable).
- **Identity**: the "Made in Bologna" footer line is part of the brand — don't strip it in a redesign.
- **Hugo version**: CI uses `latest` (currently 0.161+); local dev may be older. Avoid APIs that vary across versions (`.Site.GoogleAnalytics`, `.Site.LanguageCode`) — prefer `site.Config.Services.GoogleAnalytics.ID` and `.Lang`.
